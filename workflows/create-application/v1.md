# Create Application Workflow

Runs when a job description is provided (URL, document, or paste). All file access follows the storage-routing policy.

## Step 0 — Tracker check

When the company is not already part of the current conversation context, check the tracker first (per storage-routing: `get_pipeline()` or `application-tracker.md`) for a prior application:
1. If exactly one entry exists, proceed with that context
2. If multiple entries exist (different roles or dates), confirm with the user which position is relevant — include "new position" as an option
3. If there is no entry, treat it as a new company/JD to screen

## Step 1 — Fetch the JD

**Try WebFetch first.** If it returns a login wall (page title/body contains "sign in", "authwall", "join now", etc.) or fails:

1. Fall back to `"$PLAYWRIGHT_PYTHON" "$APP_DIR/scripts/fetch-jd.py" "<url>"`
2. Exit code 2 (auth required): tell the user to run the setup command printed to stderr — do not proceed until they confirm auth is done, then retry. Auth files live in `$APPLICANT_DIR/.auth/`. Re-run `--setup <url>` or `--import <domain>` to refresh expired cookies. If prompted for manual entry: DevTools (F12) → Application → Cookies → copy the session cookie name and value.
3. Exit code 1 (navigation error): ask the user to paste the JD text directly
4. Exit code 3 (job closed): the posting is no longer available or the position has been filled.
   - Search applications for an existing folder whose slug matches this company and role (fuzzy: lowercase folder name contains the company name and a fragment of the role title)
   - **If an existing application IS found:**
     - Update `notes.md` (per storage-routing): set `**Status:**` to `Closed` and `**Status Detail:**` to `Closed — position no longer available (YYYY-MM-DD)`; append to `## Application Log`: `YYYY-MM-DD — Position confirmed closed/no longer available via URL fetch`
     - Update tracker (per storage-routing): `update_application_status(id, 'closed', 'Position closed (YYYY-MM-DD)', null)` or move the tracker row to Rejected/Closed
     - Inform the user: "This job posting is no longer available. Application [folder] status updated to Closed."
   - **If no existing application is found:** inform the user: "This job posting is no longer available — position appears closed or filled. No further processing."
   - In both cases: stop — do not proceed with screening or document generation

**PDF and document JDs:** Use `pdftotext` or ask the user to paste if WebFetch fails.

Never stop at a step-1 failure without attempting step 2 of the fallback chain.

## Step 2 — Screen

Screen via skill `jd-evaluation` (spawn as a Haiku agent — screening is extraction, not generation). It returns: fit/no-fit with reasoning, extracted facts, best profile name.

## Step 3 — Create Application Folder (every JD, fit or no-fit)

Folder slug: `YYYY-MM-DD-company-role` — lowercase, spaces → hyphens, special characters removed.

Two JD files are required — they cannot substitute for each other; downstream tooling uses both:

- **`jd-<company>-<role-title>.[ext]`** — original JD content **verbatim** (never a summary or extraction):
  - URL source: full page text via `"$PLAYWRIGHT_PYTHON" "$APP_DIR/scripts/fetch-jd.py" --md-out <filepath-or-`-`> "<url>"`
  - PDF source: the original PDF bytes (`.pdf` extension)
  - Pasted text: the verbatim pasted content
  - **The most frequent failure mode is summarizing.** Signs you are about to: writing bullet points into `jd-*.md`, using headings like "Requirements:", constructing content from the screening output rather than the raw source. The test: `jd-*.md` should be nearly identical in length to the original page/paste.
- **`job-description.md`** — structured extraction: company, title, location, compensation, travel; role overview; required and preferred qualifications; source URL and reference to the original JD file name

Plus `notes.md` (structure below). In the notes.md JD Analysis section: record the full source URL (if URL-sourced) and the original filename.

**Save per the storage-routing policy:**

**OB1 active** (`DATA_BACKEND=ob1`) — write directly to object store, no local folder:
1. `upsert_company(name, slug)`
2. `create_application(company_name, role_title, folder_prefix='applications/YYYY-MM-DD-company-role/', profile_slug, source_url, status='pending-review')` → save UUID for later status updates
3. `upload_file('applications/<folder>/job-description.md', content, 'text/markdown')`
4. `upload_file('applications/<folder>/jd-<company>-<role>.md', content, 'text/markdown')` (or `'application/pdf'` for PDF source)
5. `upload_file('applications/<folder>/notes.md', content, 'text/markdown')`

**Fallback** (local): `mkdir "$APPLICANT_DIR/applications/<folder>/"` and write the three files there (URL source: use `--md-out "$FOLDER/jd-<company>-<role>.md"`; PDF source: copy the original PDF).

## If NO FIT (stay in Haiku)

- Create brief `notes.md` with no-fit reasoning; header must include `**Status:** Closed` and `**Status Detail:** No fit — [reason]`
- Update tracker (per storage-routing): `update_application_status(id, 'closed', 'No fit — [reason]', null)` or update the tracker Closed/Rejected section
- Stop

## If FIT (switch to Sonnet)

- Read profile content per storage-routing: `profiles/[matched-profile]/[matched-profile]-CONTENT.md` and `profiles/EXPERIENCE-REFERENCE.md`
- Generate tailored resume via skill `resume-generation`
- Create detailed `notes.md` (structure below); header must include `**Status:** Resume Ready` and `**Status Detail:** Resume generated [YYYY-MM-DD] — not yet submitted`
- Update tracker (per storage-routing): `update_application_status(id, 'resume-ready', 'Resume generated [YYYY-MM-DD] — not yet submitted', null)` or update Active Applications
- Present for user review

**After the user submits:** Run `/audit [folder-name]` to verify completeness, then `/apply "Company" "Role" "date"` to record the submission in both the tracker and notes.md atomically.

## Status changes — Two-File Rule

**Any status change** (submission, withdrawal, rejection, closing, deciding not to pursue) must update **both**:

1. The tracker — set `Status` (canonical) and `Status Detail` (free text), update Next Action, or move to Closed
2. `applications/<folder>/notes.md` — update the header `**Status:**` and `**Status Detail:**` fields to match

**Status field schema (both files):**
- `**Status:**` — one of 9 canonical values: `Pending Review` | `Resume Ready` | `Applied` | `Interview scheduled` | `Interviewed` | `Exercise/Test requested` | `Exercise/Test` | `Offer` | `Closed`
- `**Status Detail:**` — free text: dates, flags, recruiter names, context (e.g., `Applied 2026-05-15 via Greenhouse`, `Not pursuing — comp hard stop ($90K)`)

If multiple folders exist for the same role (duplicate ingest runs), update **all** of them. No exceptions for "quick" closes or decisions not to apply.

**Submission recording in notes.md:** Do NOT create a `## Submission Log` section. Update the `**Status:**` header field and append a one-line entry to the existing `## Application Log` section — the single chronological record for a folder. Format: `- [date] — Applied via [portal name] ([URL]); resume: [filename].pdf`

## JD Regeneration (Existing Applications)

When regenerating `jd-*.md` and/or `job-description.md` for an existing application folder (batch re-fetch, /ingest re-run, manual JD update):

1. After writing the updated files, compare the new `job-description.md` fit assessment against the current `notes.md` **Status** line and the tracker row.
2. If the new data reveals a status change — confirmed hard stop, closed posting, passed deadline, domain mismatch, or materially changed fit score — apply the two-file rule immediately: update `notes.md` Status/Status Detail and replace Next Steps with a Decision section; update the tracker row or move to Closed.
3. Do not close the regeneration task without checking for status drift. Regeneration without sync leaves notes.md and the tracker stale.

## notes.md Structure

Every `notes.md` must include a **Table of Contents** immediately after the header block.

### Header Block (before the TOC, fields in order)

```
**Status:** <canonical>          ← one of the 9 canonical values above
**Status Detail:** <free text>   ← full context: dates, flags, recruiter names, etc.
**Date:** YYYY-MM-DD
**Profile:** <profile-name>
**Source URL:** <url>            ← omit if no URL
```

Both `Status` and `Status Detail` must be kept in sync with the matching tracker row at all times (two-file rule).

### Required Section Order

1. Table of Contents
2. JD Analysis *(full source URL + original JD filename)*
3. Fit Assessment *(include a "Domain Connection" subsection — see the company-descriptors policy)*
4. Resume Strategy
5. Resume Evaluation Report *(JD requirement coverage table + Recruiter & ATS Appeal Analysis — see skill `resume-generation`)*
6. Company Research
7. Notes from Recruiter Interview *(add when available)*
8. Process *(hiring process steps)*
9. Interview Prep sections — **in chronological interview order** (e.g., HM interview before technical screen before panel)
10. Process Reminder *(recap of next steps)*
11. Application Log

### Interview Prep Section Rules
- Each stage gets its own H2: `## Interview Prep N — [Stage Name]`
- Sections appear in the order the interviews occur — never out of sequence
- Within each section: Logistics → Research → Talking Points → Technical Questions → Questions to Ask → Differentiators / What Not to Bring Up
